Purpose of exercise

  • Practice using keeping time-log and defect-log for Humphrey's Personal Software Process
  • Practice creating and documenting a software design
  • Practice doing proxy-based estimation of software size from historical data and conceptual design
  • Practice writing equation-based software using Dracula ACL2 mode (not Modular ACL2 mode)
  • Practice using ACL2 file i/o
  • Practice developing software testing suites
  • Practice using mechanized logic to verify software properties
  • Practice using PSP++ to generate PSP post-mortem reports
  • ABET Outcomes A, C, H, I, J, K

Project

Generate HTML files that display hand records with travelers extracted from XML records of duplicate bridge contests in such a way that, viewed with an HTML browser, hand records display in the directional format common in newspaper columns on bridge accompanied by the associated "traveler," which shows a table of results on the hand, including the scores and matchpoints recorded for each play of the hand by competing north-south and east-west pairs.

The XML schema for recording contest data to be processed in this project has some changes from the one used in previous projects, and it includes data for the traveler tables. You can download some XML files with the new schema, along with some HTML files that display hands and travelers extracted from the data.

Modules that  you may find useful in this project

  • io-utilities (Dracula teachpack)
  • list-utilities (Dracula teachpack)
  • basiclex (with extensions, downloadable)
  • xml-utilities (downloadable)

What to turn in - Note: There are two separate project submissions: iEx5dsgn (first) and iEx5 (later)

iEx5dsgn - Printed and stapled on portrait oriented sheets printed on one side only, in the following order:

  1. Software design (you can decide the form, but the design must describe basic operations, data structures, and algorithms and outline how basic operations will be composed to produce the required result)
  2. Test file (adequate sanity checks and two new predicate-based tests of non-i/o functions (new in the sense that they are not tests created for previous projects). In addition, adapt  your old tests (both sanity checks and predicate-based) to the software for this project, and include those also in your project submission.
  3. Project file (project information, predicted code objects, time log, defect log)
  4. Browser display of HTML file produced by running PSP++ on the project list and project files for iEx1, iEx2, iEx3, and iEx4dsgn

iEx5 - Printed and stapled on portrait oriented sheets printed on one side only, in the following order:

  1. Updated software design (you can decide the form, but the design must describe basic operations, data structures, and algorithms and outline how basic operations will be composed to produce the required result)
  2. Updated test file (adequate sanity checks, non-i/o functions)
  3. Predicate-test file (predicate-based tests of functions a few functions that do not perform i/o)
  4. Verification file (new theorems, not theorems submitted for a previous project) on properties that you believe your functions satisfy (mark those theorems that ACL2 does not succeed in proving with comments)
  5. Updated project file (project information, predicted code objects, time log, defect log, actual lines of code)
  6. Browser display of HTML file produced by running PSP++ on the project list and project files for iEx1, iEx2, iEx3, and iEx4
  7. File containing the program, including a function named "main" defined so that an invocation like (main "event" "handtrav" state) will write a file handtrav.htm with hand displays accompanited by travelers derived from the file event.xml
  8. Printed copies of two input files and the output files your program produces from them
